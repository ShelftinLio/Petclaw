# 验证与人工把关

AI 在本项目中承担大量生成和整理工作，但提交前需要人工把关。验证重点分为代码测试、桌面体验、安全边界、提交材料四类。

## 1. 代码测试

项目采用 Jest 测试，相关命令在 `package.json` 中定义：

```text
npm test
npm run test:ci
```

开发计划中多次要求先写失败测试，再实现，再运行目标测试。例如：

- 宠物外观：`pet-config.test.js`、`pet-appearance.test.js`
- 宠物工作室：`preload-channels.test.js`、`pet-appearance.test.js`
- 专注成长：`pet-abilities.test.js`、`pet-skills.test.js`、`pet-progress.test.js`、`focus-adventure.test.js`
- Inbox：`inbox-system.test.js`、`preload-channels.test.js`、Renderer 合约检查
- 亲密度：`pet-affinity.test.js`、`pet-progress.test.js`、`preload-channels.test.js`

这些测试计划可在 `evidence/superpowers-plans` 中查看。

## 2. 静态检查

对关键 JavaScript 文件使用 `node --check` 做语法检查。实现计划中出现的检查对象包括：

```text
node --check main.js
node --check preload.js
node --check pet-affinity.js
node --check pet-progress.js
```

## 3. 桌面端人工验收

人工验收关注：

- Electron 应用能否正常启动。
- 宠物窗口、工具栏、字幕窗口和设置窗口是否正常显示。
- Pet Studio 能否打开、预览宠物并切换资源。
- 专注冒险能否开始、结束并结算奖励。
- Inbox 是否能收纳文件、剪贴板文本、链接和图片。
- 亲密度反馈是否不会干扰正常工作。
- Gateway、日志、模型切换等原有功能没有被破坏。

## 4. 安全与隐私检查

提交前检查：

- 不提交 `.env`、API Key、token、用户本地凭据。
- 不提交 `pet-config.json` 中的私人运行配置。
- 不把运行缓存、临时文件、日志密钥或本地账号信息放入使用记录包。
- evidence 目录只复制设计文档、实现计划、git 摘要和图片生成记录。

## 5. 本 AI 使用记录包的校验方式

本包生成后检查：

- ZIP 文件能正常打开。
- 根目录包含 4 份 Markdown 主文档。
- `evidence/` 包含 specs、plans、imagegen 和 git 日志。
- 压缩包没有包含 `node_modules`、`.git`、`.env`、运行缓存或私密配置。

## 6. 推荐提交话术

```text
AI 使用记录已打包为 zip。包内包含 Prompt 设计、迭代思路、生成内容占比说明，以及 Codex 设计/计划文档、git 日志和 AI 图片生成 Prompt 等证据材料。
```
